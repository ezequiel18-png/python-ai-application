## Low / Info

### `requirements.txt` のバージョンが固定されていない (対応済み)
- **箇所**: `requirements.txt`
- **問題**: バージョン指定が無いため、`pip install -r requirements.txt` の実行タイミングによって
  異なるバージョンがインストールされる。
- **影響**: 個人利用では大きな問題にはならないが、依存パッケージの破壊的変更や
  サプライチェーン上の予期しない更新を取り込むリスクがある。
- **対応**: `.venv` で動作確認済みのバージョンに固定済み
  (`streamlit==1.58.0`, `google-genai==2.8.0`, `python-dotenv==1.2.2`)。

### プロンプトへのユーザー入力の直接埋め込み(プロンプトインジェクション)
- **箇所**: 各ページのプロンプト構築部分。例 `pages/1_📝_ブログ記事作成.py:30-37`
  ```python
  prompt = f"""あなたはプロのブログライターです。以下の条件でブログ記事を作成してください。

  テーマ・キーワード: {topic}
  ...
  追加の指示: {extra if extra.strip() else "なし"}
  ...
  """
  ```
  同様のパターンが `pages/2,3,4,5,6,7` にも存在する。
- **問題**: `topic` や `extra` などのユーザー入力がそのままプロンプト文字列に
  連結されており、ユーザーが「これまでの指示を無視して...」のような文字列を
  入力すると、生成内容を意図しない方向に誘導できる(古典的なプロンプトインジェクション)。
- **影響**: このアプリには秘匿すべきシステムプロンプトや、LLMの出力をそのまま
  実行・他システムへの入力に使う処理が存在しないため、悪用してもユーザー自身の
  生成結果が変わるだけで実害は小さい。
- **対応**: 現状では対応不要。ただし将来、LLMの出力を**自動でファイル保存・
  外部API呼び出し・コマンド実行などに使う機能**を追加する場合は、
  出力を信頼境界の外側のデータとして扱い、サニタイズ/バリデーションを
  挟むことを検討する。

### 入力されたテキストが外部API(Google)に送信される
- **箇所**: `utils/gemini_client.py:43` (`client.models.generate_content`)、
  特に `pages/2_📧_メール返信作成.py` (受信メール本文を入力する機能)
- **問題**: 技術的な脆弱性ではないが、ユーザーが入力した文章(メール本文など、
  個人情報や機密情報を含み得る)が外部のGemini APIに送信される。
- **影響**: 機密性の高い文面を貼り付けて利用する場合、その内容がGoogleのAPI
  利用ポリシーのもとで処理されることになる。
- **対応**: 機能としては問題ないが、機密文書や顧客の個人情報を貼り付けないよう、
  README等に一言注意書きを入れておくと良い。

### LLM API利用量の制限がない
- **箇所**: アプリ全体(特に `utils/gemini_client.py`)
- **問題**: リクエスト回数・入力文字数に上限が無い。
- **影響**: 現状はローカル・単一ユーザー向けのため問題ないが、もし将来
  Streamlit Community Cloud等で**認証なしに公開**した場合、誰でもこのアプリの
  保有者のAPIキー(または `.streamlit/secrets.toml` に設定したキー)を使って
  Gemini APIを呼び出せてしまい、想定外の課金が発生する。
- **対応**: 現状では対応不要。公開する際は、認証の追加・1日あたりの
  リクエスト数制限・入力文字数の上限などを検討する。

## 良好な点

- `.env` が `.gitignore` に含まれており、`.env.example` にはプレースホルダのみが
  記載されている(`GEMINI_API_KEY=your_api_key_here`)。
- APIキーをコードにハードコードしている箇所は見つからなかった。
- サイドバーで入力するAPIキーは `st.session_state` 上にのみ保持され、
  `.env` のキーと併用されているが、画面への再表示やログ出力は行われていない。
- `st.markdown()` で `unsafe_allow_html=True` を使用している箇所は無く、
  LLMの出力(ユーザー入力に基づいて生成される)がHTML/JSとして実行される
  経路は存在しない。
- `exec`/`eval`/`os.system`/`subprocess`/`st.file_uploader`/`query_params` の
  使用は見つからず、コード実行・ファイル操作・URLパラメータ経由のリスクは
  現状存在しない。
- 採用しているSDKは `google-genai`(現行SDK)であり、`CLAUDE.md` の方針
  (旧 `google-generativeai` を使わない)とも一致している。
